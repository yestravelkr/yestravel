# 주문 이력 추적

## 목적

주문의 생명주기 전체를 추적하여 관리자가 주문 상태 변경 이력을 투명하게 확인할 수 있도록 한다. 누가(Actor), 언제, 어떤 이유로 주문 상태를 변경했는지 기록한다.

## 핵심 기능

| 기능 | 설명 | 사용자 관점 |
|------|------|------------|
| 주문 이벤트 자동 기록 | 결제/상태변경/클레임 발생 시 자동 이력 생성 | 주문 이력이 자동으로 누적됨 |
| 이력 조회 | 주문별 시간순 이력 목록 제공 | 관리자가 주문 상세에서 이력 확인 |
| 옵션별 필터 | 특정 옵션(상품)에 대한 이력만 필터링 | 복수 옵션 주문에서 특정 상품 추적 |
| 액터 식별 | 시스템/고객/관리자 구분 기록 | 누가 변경했는지 명확히 확인 |

## 사용자 흐름

### 이력 자동 기록 (백그라운드)

1. 고객이 결제 완료 → "주문 생성" + "결제 완료" 이력 자동 기록
2. 고객이 취소/반품 요청 → "취소 요청" 또는 "반품 요청" 이력 기록
3. PAID 상태 취소 요청 → 자동 승인되어 "자동 취소 승인" + "환불 처리" 이력 기록
4. 관리자가 상태 변경 → "상태 변경" 이력 기록 (이전/이후 상태 포함)
5. 관리자가 클레임 승인/거절 → "취소 승인"/"반품 승인" 등 이력 기록

### 이력 조회 (관리자)

1. 백오피스 주문 상세 페이지 접속 → "주문 이력" 버튼 클릭
2. 모달에서 시간순 이력 목록 확인
3. 옵션별 필터 칩으로 특정 상품 이력만 조회

## 비즈니스 규칙

| 규칙 | 설명 |
|------|------|
| 자동 기록 | 주문 상태 변경 시 항상 이력이 자동 기록됨 |
| 실패 허용 | 이력 기록 실패가 주문/결제 처리를 방해하지 않음 |
| 클레임 추적 | 상태 변경 없는 클레임 이벤트도 감사 목적으로 기록 |
| 환불 분리 기록 | 취소 승인과 환불 처리를 별도 이력으로 기록 |
| 3가지 액터 | 시스템(자동처리), 고객(요청/철회), 관리자(상태변경/클레임처리) |
| PurchaseLog와 역할 구분 | PurchaseLog는 금액 변동(정산용), OrderHistory는 이벤트 이력(감사용) |
