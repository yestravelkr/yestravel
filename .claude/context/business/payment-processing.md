# 결제 처리

## 목적

고객이 상품을 선택하고 결제를 완료하면, 임시 주문을 정식 주문으로 확정하고 PG사(포트원)를 통해 결제를 승인한다. 관리자가 주문을 취소할 경우 PG사를 통해 전액 또는 부분 환불을 처리한다.

## 핵심 기능

| 기능 | 설명 | 사용자 관점 |
|------|------|------------|
| 결제 완료 처리 | 임시 주문을 정식 주문으로 변환하고 PG 승인 | 고객이 결제 버튼 클릭 후 주문이 확정됨 |
| 주문 타입별 처리 | 호텔/배송/E-티켓에 따라 적절한 주문 생성 | 상품 유형에 관계없이 동일한 결제 경험 |
| 결제 취소 | PG사를 통한 전액/부분 취소 + 호텔 재고 복구 | 주문 취소 시 자동 환불 + 재고 원복 |
| 결제 이력 관리 | Payment 엔티티로 결제 상세 정보 보존 | 고객/관리자가 결제 내역 확인 가능 |
| 호텔 재고 관리 | 결제 시 숙소 날짜별 재고 자동 차감 | 재고 부족 시 결제 자동 취소, 초과 예약 방지 |
| 동시 결제 경합 처리 | 동일 객실 동시 예약 시 선착순 확정 | 먼저 결제한 고객이 예약 확정, 나머지는 자동 환불 |
| 결제 완료 알림톡 | 호텔 결제 완료 시 카카오 알림톡으로 예약 안내 발송 | 고객이 예약 접수 확인 및 예약 확정 안내를 받음 |

## 사용자 흐름

### 결제 완료

1. 고객이 상품 선택 후 결제 진행 -> 임시 주문(TmpOrder) 생성
2. PG사(포트원) 결제창에서 결제 수행
3. 결제 완료 시 시스템이 임시 주문을 정식 주문으로 변환
4. PG사에 결제 승인 요청 -> 승인 완료 -> 결제 정보 저장
5. 호텔 상품인 경우, 카카오 알림톡으로 예약 안내 메시지 발송 (주문 정보, 예약 확인 링크, 변경/취소 안내 포함)
6. 고객에게 주문 완료 안내

### 결제 취소

1. 관리자 또는 고객이 주문 취소 요청 (전액 또는 부분)
2. 호텔 상품인 경우 예약된 날짜의 재고를 자동 복구
3. 시스템이 PG사에 취소 요청 전송
4. PG사 취소 승인 -> 고객에게 환불 처리

## 비즈니스 규칙

- 결제 승인이 완료된 후 결제 정보 저장에 실패하더라도, 이미 승인된 결제는 취소하지 않는다 (데이터 정합성보다 결제 안정성 우선)
- 주문번호(orderNumber)는 임시 주문 ID를 기반으로 인코딩되어 PG사와 통신에 사용된다
- 이미 결제된 요청이 중복 도달하면 TmpOrder가 이미 삭제된 상태이므로 NotFoundException을 반환한다 (멱등성 - 중복 주문 생성 방지)
- 호텔 결제 완료 시 알림톡 발송에 실패하더라도 결제 자체는 정상 완료된다 (알림 실패가 결제를 실패시키지 않음)
- 알림톡 발송 실패 시 LMS(장문 문자)로 대체 발송을 시도한다
- 알림톡은 호텔 상품에 대해서만 발송되며, 배송/E-티켓 상품에는 발송되지 않는다

### 호텔 재고 관리 규칙

- 재고는 **숙소(product) + 날짜** 단위로 관리된다. 같은 숙소의 서로 다른 객실 옵션은 동일한 재고를 공유한다
- 다른 날짜 또는 다른 숙소는 재고가 독립적이다
- 여러 날짜에 걸친 주문은 모든 날짜의 재고가 충분해야 한다. 1개 날짜라도 재고 부족 시 전체 결제가 취소된다 (all-or-nothing)
- 재고 부족으로 결제 실패 시 PG사에 자동 취소 요청을 보내고, 해당 주문은 CANCELLED 상태가 된다
- 동시 결제 경합 시 DB 비관적 락으로 선착순 처리하며, 재고 초과 결제는 자동 취소/환불된다

### 호텔 재고 복구 규칙

- 주문 취소 시 호텔 상품이면 예약 날짜의 재고가 자동으로 1씩 복구된다
- 재고 복구는 3개 취소 경로(백오피스 클레임, 백오피스 주문 환불, 고객 클레임) 모두에서 동일하게 동작한다
- 호텔이 아닌 상품(배송/E-티켓)의 취소에서는 재고 복구가 발생하지 않는다
- 재고 복구도 비관적 락으로 보호되어 동시 취소 시에도 정합성이 유지된다
